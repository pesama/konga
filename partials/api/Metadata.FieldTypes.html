FieldTypes
service in module Metadata

Description

Field types configure the appeareance of the input on the forms.

Data types vs. Field types

Data types define structure, while field types define rendering. Depending on the Data type of the field there would be different eligible field types to choose.

Based on the Data types there is also a default field type to choose, though you can choose any other to adapt it to your favorite appeareance.

  • BOOLEAN

    Default input for boolean values. It's intended for Yes/No fields. It renders two radio buttons as layout, one for true and one for false. On search mode it renders to checkboxes instead, for allowing selecting multiple/none options. On results table value is mapped and render 'Yes' for true and 'No' for false.

    Data types boolean

    The boolean input is designed to be used with boolean data types.

    The boolean input uses .

    Multiplicities

    The boolean input is designed for ONE multiplicity.

    Configuration

    Inherited from rawInput

    The boolean input does not extend nor override raw functionality.

    Events

    Inherited from rawInput

    The boolean input does not fire nor listen further events.

    Validations

    The boolean input allows these validations:

    • required: The input won't be valid until it's some value it's filled. Although this works fine, on boolean inputs you should consider using defaults value.

  • CALENDAR
    

    The calendarInput is a rawInput field type that renders a fullcalendar as field's layout, if the field's fieldType is set to CALENDAR.
    

    The calendar field is just a container of objects, that have properties for defining an Event Object. The calendar takes these properties and renders the events into the ui, giving you full access to your events - and the calendar for including new - through events.
    

    Data types calendar
    

    The calendar input is designed to be used with calendar data types.
    

    The calendar input uses . 
    

    Multiplicities
    

    The calendar input is designed for MANY multiplicity.
    

    Configuration
    

    Inherited from rawInput
    

    The calendar input does not extend nor override raw functionality.
    

    Events
    

    Inherited from rawInput
    

      

      • 

      • calendar-day-clicked: Fired when the user clicks on a calendar day.
        • 

      

      • 

    • calendar-event-clicked: Fired when the user clicks on an existing event.
      • 

    • calendar-event-render: Used when an event finishes its rendering process.
      • 

    

    All calendarInput events include a data object with the information about the action:
 
       {
     date: Date,
     jsEvent: Event,
     view: view,
     resourceObj: resourceObj
   }
 
    

    Validations
    

    The calendar input allows these validations:
    

      

    • [min|max]Length: The input won't validate unless the event size is within the bounds defined.
      • 

    

    For the calendar input you might want to add custom actions to allow easier interactions with the calendar and its events.

  • COMPLEX
    

    Complex fields renders configured inner fields instead of the field itself (e.g. if you want to render Vehicle properties (such as brand and wheel number) instead of the vehicle itself). You need to declare into the field's showInUpdate->fields attribute the fields to render. 
    

    

    • 

  • CSS
    

    Renders a container with a css class assigned with the value of the field. Useful for inserting colored bullets, icons to categorise...
    

    

    • 

  • CUSTOM
    

    Custom input. If none of the existing field types suit your needs, you can tag the field as CUSTOM and provide a valid mapped value to a view with your custom view. it's done via CUSTOM_FIELD_TYPE property.

  • PRICE
    

    Numerical input, for price representation. It's basically a NUMBER input, but with currency rendering. It relies on a CURRENCY configuration param to setup the currency, and a CURRENCY_SOURCE to define where this currency comes from (useful for varying currencies depending on the entity).
    

    

    • 

  • QUANTITY
    

    It's basically a numerical input, but for referencing quantities. It has several configuration options to setup the unit, the decimal amounts, etcetera:
    

      

    • QUANTITY_UNIT: Unit.
      • 

    • QUANTITY_UNIT_SOURCE: Where to get the unit from.
      • 

    • QUANTITY_DECIMAL: Decimal no.
      • 

    • QUANTITY_DECIMAL_SOURCE: Where to get the decimal no. configuration from.
      • 

    

    

    • 

  • SELECT
    

    Used for COMPLEX data-typed fields. Text input with a button for quick targeting eligible elements. This field type uses all complex-intended parameters defined on the data-type. Supports both multiplicities.
    

    

    • 

  • SWITCH
    

    A switch is a physical toggle-button for selecting boolean values. 
    

    

    • 

  • TABLE
    

    Renders a table to show data. These tables are intended to render numerical data, range-based, depending on several configuration parameters given to the field's type:
    

      

    • TABLE_CELL_FILTER: Defines the filter to apply for each cell's value.
      • 

    • TABLE_CONF_X_AXIS_MIN: Defines the minimum x-axis value.
      • 

    • TABLE_CONF_X_AXIS_MAX: Defines the maximum x-axis value.
      • 

    • TABLE_CONF_X_AXIS_PROPERTY: Defines the property (whitin the inner's table entity) to render onto the x-axis.
      • 

    • TABLE_CONF_X_AXIS_STEP: Defines the numerical step between one column and the next.
      

      • 

    • TABLE_CONF_Y_AXIS_MIN: Defines the minimum x-axis value.
      

      • 

    • TABLE_CONF_Y_AXIS_MAX: Defines the maximum x-axis value.
      • 

    • TABLE_CONF_Y_AXIS_PROPERTY: Defines the property (whitin the inner's table entity) to render onto the x-axis.
      • 

    • TABLE_CONF_Y_AXIS_STEP: Defines the numerical step between one row and the next.
      •
